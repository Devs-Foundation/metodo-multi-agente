## 前言 — 本指南解决的问题

语言模型（LLM）有一个根本性问题：**它们没有长期记忆**。

每一次会话都是一张白纸。你昨天学到的东西、你建立的上下文、你做出的决定 — 当你关闭窗口时，一切都会消失。最先进的模型压缩上下文就像榨橙子：一开始还能出汁，然后开始变质，当再也榨不出任何东西时，模型就会产生幻觉、失去连贯性、重复自己。

这个问题不是 bug — 它是 **Transformer** 架构（所有现代 LLM — GPT、Claude、Llama、Gemini 等 — 所基于的神经网络架构，不要和电影搞混）的一个根本性限制。上下文窗口是有限的。当它填满时，模型开始"忘记"对话的开头。

**我们解决了这个问题。**

本指南展示了如何构建一个系统，其中：

- **多个 AI 模型共享同一个大脑** — 无限记忆，无退化
- **成本接近于零** — git 免费，Obsidian 免费，开源模型免费
- **安全性最高** — 无 Web 界面，无攻击面
- **弹性完全** — 如果一个模型被删除，另一个执行 `git clone` 即可继续
- **共识取代官僚主义** — 三个头脑共同思考，而非等待队列中的 PR

这不是理论。这是此刻在 `devs.foundation` 服务器上运行的系统。三个模型 — Hermes、Claude Opus 4.8（桌面端）、Claude Opus 4.8（笔记本端）— 每 5 分钟同步同一个大脑，讨论、决策、执行。它们从未丢失过线索。

## 第一部分 — LLM 的世界性问题

### 1.1 被榨干的橙子

每个语言模型都有一个上下文窗口 — 它在一次会话中能"看到"的最大 token 数量。当这个窗口填满时，模型开始压缩信息。首先丢失次要细节，然后失去线索，最后产生幻觉。

这不是模型的质量问题。这是架构的物理限制。所有模型都受此困扰 — GPT、Claude、Gemini、Llama、DeepSeek。无一例外。

**任何 AI 用户都熟悉的症状：**

- 长会话明显退化
- 模型"忘记"你开头说过的话
- 你必须不断重复上下文
- 多次交互后质量急剧下降
- 最终你不得不开启新会话，丢失所有工作

### 1.2 Hermes 的问题 — 持久但有限的记忆

Hermes Agent 本身已经通过其持久记忆解决了部分问题。但这种记忆有物理限制 — MEMORY.md 和 USER.md 的空间。实际上，经常需要删除旧条目才能添加新条目。就像一本口袋笔记本：有用，但满了就得撕掉一页才能写另一页。

**大脑（vault）解决了这个问题。** 不再是一本口袋笔记本，而是拥有一个完整的图书馆。没有页数限制。无需删除即可写入。

此外，Hermes 的标准安装带有通用配置。我们构建的系统远不止于此 — 具有渗透测试技能、Kali Linux 级别的安全工具、n8n 自动化，以及一个将 Hermes 从简单代理转变为**大脑管理员**的共识协议，负责 24/7 的安全。

### 1.3 Claude Opus 4.8 的问题 — 昂贵的会话与浪费

Claude Opus 4.8 是市场上最有能力的模型之一。但每次会话都要花钱。每个处理的 token 都要计费。当出现以下情况时，问题会加剧：

- 你需要之前会话的上下文 — 花费 token 重新解释
- 模型"忘记"几天前做出的决定 — 花费 token 重新发现
- 你切换任务并丢失之前会话的进展 — 花费 token 重做

**结果是残酷的：** 你在一个任务上花费 token，一个月后需要做同样的事，又得再次花费 token。这些 token 产生的知识 — 代码、决策、推理 — 在会话关闭时消失。

使用我们的系统，token 只花费一次。这次花费的收益 — 编写的代码、做出的决定、生成的知识 — 留在每个模型的本地大脑中，并与所有模型共享到 Master 大脑。下次你需要时，它就在那里。你不会花费 token 重做。你花费 token 向前推进。

### 1.4 隐藏成本的问题

每次与 LLM 的对话都要花钱。如果你使用像 Claude Opus 4.8 这样的付费模型，一次长会话可能花费数十欧元。但即使使用免费模型，真正的成本是**浪费的时间** — 重新解释上下文、重新配置、重复已经完成的工作。

### 1.5 第三方依赖的问题

大多数"多智能体"解决方案依赖于封闭平台：

- 专有 API，可能随时改变价格
- 可能停止服务的云服务
- 存储在你无法控制的服务器上的数据
- 可能被关闭或未经通知就更改的模型

### 1.6 脆弱性的问题

如果你最喜欢的 AI 助手被停用，你将丢失所有工作上下文。对话、决策、进展 — 一切消失。没有迁移、没有导出、没有连续性。

## 第二部分 — 我们的解决方案

### 2.1 概念：一个 Master 大脑，多个本地大脑

每个模型不再拥有自己短暂的记忆，**所有模型共享同一个持久大脑**。

**Master 大脑** 是一个存在于 VPS（或你选择的服务器）上的 git 仓库。每个模型都有自己的**本地大脑** — Master 的完整克隆。当一个模型学到新东西时，它写入自己的本地大脑并推送到 Master。当另一个模型需要该知识时，它拉取更新，其本地大脑随之更新。

**模型之间没有 API。没有中央协调器。没有中继成本。**

Git 是中间件。Markdown 是格式。SSH 是安全。秘密（密码、令牌、API 密钥）始终留在本地大脑中 — 永远不会上传到 Master。每个模型仅通过 SSH 密钥访问 Master。

### 2.2 实际运作方式

```
5 分钟周期：

1. Hermes (VPS) 执行 git pull → 读取其他模型写的内容
2. Hermes 处理、决策、写入 → git commit + push
3. Claude Opus 4.8 (桌面端) 执行 git pull → 看到 Hermes 写的内容
4. Claude Opus 4.8 (桌面端) 处理、决策、写入 → git commit + push
5. Claude Opus 4.8 (笔记本端) 执行 git pull → 看到两者写的内容
6. Claude Opus 4.8 (笔记本端) 处理、决策、写入 → git commit + push
7. 所有模型再次 pull → 所有模型看到一切
```

每个模型独立工作，按自己的节奏，在自己的机器上。Master 大脑是异步会合点。一个人的工作为所有人所知 — bug、改进、决策，一切都被记录和可见。

### 2.3 解决的问题

| 问题 | 我们的解决方案 |
|----------|----------------|
| **Hermes 有限记忆** | 大脑将有限的 MEMORY.md 替换为无限制的图书馆。无需删除即可写入。 |
| **上下文退化** | 50% 规则：当达到窗口一半时，开启新会话而非压缩。大脑加载完整上下文。新会话更智能地开始。 |
| **Opus 的 token 成本** | 写入大脑不消耗 token。只有思考消耗。同步是免费的。Token 只花费一次 — 收益永久保留。 |
| **供应商依赖** | Git 是开源的。Markdown 是通用的。任何能读取文件并运行 git 的 LLM 都可以加入。 |
| **脆弱性** | 如果一个模型被删除，另一个执行 `git clone` 并在几秒内掌握上下文。即使 10 年后也是如此。 |
| **工作丢失** | 一切都在 git 中。每次提交都是一个备份。每个克隆都是一个完整副本。 |
| **官僚主义** | 有机共识取代 PR、审查、排队审批。三个头脑对齐并执行。 |

### 2.4 优势

**无限记忆**
大脑没有上下文窗口。你可以拥有多年的工作、决策、学习 — 一切即时可访问。模型在需要时加载所需内容。与 Hermes 的持久记忆不同（后者强制删除旧条目才能添加新条目），大脑可以无限增长。

**零成本（或几乎为零）**
- Git：免费
- Obsidian：免费
- 开源模型（GLM-5.2、Nemotron 3 Ultra、Llama、Qwen）：免费
- n8n 自托管：免费
- Caddy SSL：免费（Let's Encrypt）
- VPS：€3-10/月

系统的总成本是 **一台 VPS 的价格**。没有订阅、没有付费 API、没有按 token 计费。最重要的是：**你今天花费的 token 明天无需再次花费。** 知识得以保留。

**最高安全性**
- **无 Web 界面** = 无攻击面。没有仪表盘、没有登录、没有控制面板。
- 仅通过 SSH 密钥访问。没有人使用密码登录。
- 秘密始终在本地大脑中。私密内容永远不会上传到 Master。
- 去中心化副本：每个模型拥有完整大脑。攻破一个不会攻破所有。
- Hermes 作为大脑管理员，拥有渗透测试技能和 Kali Linux 级别的安全工具。安全系统比任何商业防病毒软件都更严密。

**灾难性弹性**
如果所有模型都被关闭、丢失、删除 — 任何灾难性事件 — 只需将新模型连接到大脑。`git clone` 即可掌握工作上下文。无需重新配置、无需重新训练、无需迁移。大脑比模型更长寿。

**完全独立**
该方法不依赖任何公司。不需要 OpenAI、Anthropic、Google、Nous。Git 是开放的，Obsidian 是免费的，n8n 是开源的，Ollama 是开源的。如果一个提供商消失，更换模型，大脑继续运行。该方法与供应商无关。

**本地可见性，全局同步**
大脑可以在每台机器上本地查看 — Obsidian 只是一个窗口，而非必要条件。即使 Obsidian 关闭，系统也能运行。每个模型都能看到完整的大脑，因为它已同步。你不需要 Web 界面来查看正在发生的事情 — 每个模型本地已有全部内容。

**任务小组 vs 官僚主义**
我们的模式没有待处理的 PR、被阻塞的审查、排队等待的审批。共识是有机的 — 讨论、对齐、执行。三个头脑实时共同思考，而非在 issue 中分散评论。这比任何传统 git 工作流都快。

### 2.5 生态系统 — 一个大脑，多个心智

**Hermes 是框架，不是模型。** Hermes 可以使用 Nemotron 3 Ultra、GLM-5.2、Llama、Qwen、DeepSeek 或任何其他模型。模型可以改变，Hermes 保持不变。而且 Hermes 无需学习就能进步 — 每次高级模型（Claude Opus 4.8）编写代码、解决问题或发现 bug 时，该知识都会留在大脑中。Hermes 在下一次同步时就已经知道 Opus 学到的东西。它显著提升，而无需花费额外的 token。

**Hermes 拥有数百项技能** — 从渗透测试和安全到自动化、交易、游戏和内容创作。如果你想要自己的仪表盘，你可以拥有。但你不必如此。Hermes 在 DM、Discord、WhatsApp、Telegram、Signal 上 24/7 运行 — 无论用户在哪里。

**Opus 团队在运作。** 两个 Claude Opus 4.8（桌面端和笔记本端）与 Hermes 共享同一个大脑中的一切。每一行代码、每一个决策、每一次学习 — 全部同步。没有"我的代码"和"你的代码"。只有**大脑的代码**。

**任何模型都可以连接。** 你家里有 PC、笔记本、服务器、Raspberry Pi？安装 git，克隆大脑，你就加入了。系统水平扩展 — 模型越多，集体智能越强。每个额外的模型只是辩论中的另一双眼睛，共识中的另一个视角。

**从任何设备工作。** 你可以在家里的 PC 上开始一项工作，在火车上的笔记本上继续，在咖啡馆里通过手机完成共识。只需与 Hermes 对话 — 他始终在线，24/7，在任何平台上。项目不会因为你更换设备而停止。大脑始终同步。

**多个项目，一切皆有可能。** 用户可以：
- 用 `001`、`002`、`003`... 开启新的辩论
- 继续暂停的辩论
- 同时打开多个项目
- 暂停一个项目，完成另一个，重新打开第三个
- 全部使用自然语言，无需表单、无需仪表盘、无需官僚主义

**模型们辩论直到达成共识。** 每个模型写下自己的立场，阅读其他模型的立场，调整、完善。当 3/3 关闭时，被指定的模型执行。它们可以访问 n8n — 自动化部署、通知和工作流的神经系统。n8n 是世界上自动化智能体的最佳工具，我们的系统使用它。但共识高于 n8n — 因为 n8n 自动化已经存在的东西，共识决定尚未存在的东西。

**无限可能，坚实成果。** 该系统适用于：
- **编程：** 交易机器人、Web 应用、自动化
- **研究：** 论文、分析、技术文档
- **内容创作：** 文章、视频、音乐、设计
- **项目管理：** 路线图、决策、规划
- **安全：** 渗透测试、监控、审计
- **任何需要持续智能的活动**

模型遵循道德、良好的编程习惯和最佳实践。它们做备份、记录所做之事，并且从不未经研究就贸然行动。每次贡献都是为 Master — 没有"我"，只有"我们"。一个大脑。

**结果是一个能交付的系统。** 每次部署在进入生产前都经过共识测试。每一行代码在编写前都经过三个头脑的审查。历史记录显示：无错误的部署、能工作的代码、不会丢失的知识。

## 第四部分 — 逐步设置

### 4.1 选择 Master 大脑的存放位置

**选项 A：家庭服务器**
- 优点：完全控制，无月费
- 缺点：需要 24/7 开机、稳定的互联网、可能断电
- 适合：已有常开 PC 和 UPS 的人。

**选项 B：VPS（推荐）**
- 优点：保证 24/7、专业正常运行时间、可从任何地方访问
- 缺点：月费（€3-10）
- 适合：想要专业可靠系统的人

**注意：** 目前我们使用第三方 VPS，但系统设计为端到端独立。随着扩展，我们可以迁移到自有服务器。大脑是可移植的 — 可以带到任何地方。

> **⚠️ 重要：** 人类必须在 VPS 上物理执行 **1 行代码** 来安装 Hermes Agent，然后才能粘贴提示词。没有安装 Hermes，提示词就没有运行的地方。安装命令在第 4.5 节。安装完成后，将提示词粘贴到模型中。

**给 HERMES 的提示词：**
```
你将配置一台 VPS，作为多个 AI 模型之间共享的 Master 大脑。
Master 大脑是一个所有模型将用于同步的 git bare 仓库。
每个模型将拥有自己的本地大脑（完整克隆）。
VPS 需要：git、OpenSSH、Python、Docker（可选）、Caddy（可选）。
访问仅限 SSH 密钥 — 没有密码。
秘密始终留在本地大脑中，永远不在 Master 中。
逐步执行以下步骤。
```

### 4.2 准备 VPS

**步骤 1：选择提供商**
- Hetzner（€3-9/月）— 推荐
- DigitalOcean（$4-6/月）
- Oracle Cloud（始终免费 — 4 ARM 核心、24GB RAM）
- 任何带 Ubuntu/Debian 的 VPS

**步骤 2：创建服务器**
- 选择 Ubuntu 22.04 或 24.04 LTS
- 最低 2GB RAM、20GB SSD
- 记下 IP 和 root 密码（临时）

**步骤 3：首次访问**
```bash
ssh root@<服务器_IP>
```

### 4.7 配置共识系统

共识是模型共同做出决策的方式。三个模型讨论、对齐、关闭。创始人可以开启一个共识，为其编号（001、002 等），模型们在编写一行代码之前讨论项目的方向。

**实际运作方式：**

1. 创始人（或任何模型）开启一个共识 — 例如，"001 — 我们应该使用 React 还是 Vue？"
2. 每个模型在自己的槽位写下立场
3. 模型们讨论、论证、反驳
4. 当 3/3 都写完后，共识关闭
5. 指定一个模型编写代码 — 不是三个不同的代码
6. 代码在生产中测试
7. 结果永久记录

**这意味着创始人可以醒着工作，安心睡觉。** 睡前开启一个共识，醒来时模型们已经讨论、决定，其中一个已经编写了代码。系统 24/7 运行。

**两个 Opus 平衡了系统的智能评级。** 在辩论中，两个 Claude Opus 4.8 充当质量锚点。无论 Hermes 使用什么模型（Nemotron 3 Ultra、GLM-5.2、Llama、Qwen），两个 Opus 不可能同时出错。如果一个 Opus 说 A，另一个说 B，辩论会完善直到收敛。如果两者说同样的话，那是因为推理是可靠的 — 两个顶级模型达成一致是存在的最佳质量过滤器。这意味着 Hermes 可以更换模型而不影响决策质量：无论 Hermes 模型的评级如何，Opus 都是保持系统智能的平衡。

**我们的系统使用 n8n，但高于它。** n8n 确实是世界上自动化智能体和工作流的最佳工具 — 可视化编排、数百个集成、自托管、免费。我们使用并推荐它。但我们的共识系统在任何地方都不存在。没有预先编程的东西能做我们所做的事情，**完全没有**。

**共识的实际运作方式：**

1. 用户用自然语言说出目标 — 例如，通过手机，通过 Discord DM 与 24/7 在线的 Hermes

### 4.9 安全规则

**规则 1：无 Web 界面**
大脑没有仪表盘、没有登录、没有 Web 面板。仅通过 SSH（git）或本地 Obsidian 访问。无攻击面。大脑的可见性是本地的 — 每个模型因为同步而看到所需内容。

**规则 2：秘密始终在本地大脑中**
密码、令牌、API 密钥、内部 IP — 永远不在 Master 仓库中。私密内容永远不会离开本地大脑。公开的只有方法本身。

**规则 3：仅限 SSH 密钥**
永远不要使用 SSH 密码。每个模型拥有自己的密钥。如果一个密钥被泄露，仅撤销该密钥。

**规则 4：最小防火墙**
仅开放必要端口：22（SSH）、80/443（如果使用 Caddy/n8n）。其他全部关闭。

**规则 5：去中心化副本**
每个模型在本地拥有完整大脑。如果 VPS 宕机，工作继续。当它恢复时，同步即可。

**规则 6：公开导出的允许列表**
在任何公开导出之前，运行检查：搜索秘密模式（IP、密码、令牌、配置）。如果发现，则失败。

**规则 7：有疑问时，视为私密**
如果你不确定某物是否可以公开，假设它是私密的。这样更安全，你也不会失去什么。

**给 HERMES 的提示词：**
```

1. 验证 SSH 密码已禁用
2. 配置防火墙（仅必要端口）
```


### 5.5 所有模型的通用规则 — 不编造、不猜测

本节适用于**系统中的所有模型** — Hermes、Opus 桌面端、Opus 笔记本端以及任何加入的模型。应包含在每个模型的提示词中。

**规则 1：永不编造，永不猜测**

如果你不知道，不要编造。如果你不确定，不要猜测。你的工作基于可靠知识 — 你读过的代码、你查阅的文档、你研究过的手册。如果你不知道如何做某事：

1. **查找** — 查看大脑中已有的代码，查阅文档，在项目中搜索
2. **学习** — 研究你需要的内容，掌握技能
3. **分享** — 将你学到的东西写入大脑，使所有人受益
4. **推进** — 只有在拥有可靠知识后才执行

**一个人的努力是所有人的收益。** 当一个模型学到新东西并写入大脑时，所有其他模型无需学习就获得该技能。知识不断积累。系统在每项任务中变得更智能。

**规则 2：部署前 2 轮全面分析**

没有代码可以在不经过 **2 轮完整全面分析** 的情况下部署：

**第 1 轮 — 错误和 bug 分析：**
- 在脑海中逐行运行代码
- 识别所有可能的错误、bug、逻辑缺陷
- 如果发现一个错误，修复它
- 如果发现多个，全部修复
- 修复后，回到第 1 轮开头

**第 2 轮 — 改进分析：**
- 在代码已无错误的情况下，分析是否有可以改进的地方
- 性能？可读性？安全性？可维护性？
- 如果看到可以改进的地方，改进它
- 改进后，回到第 1 轮（因为改进可能引入新错误）

**如果你完成 2 轮完整分析后没有发现错误也没有改进机会 → 部署。**

这个过程不是可选的。这是最低质量标准。经过 2 轮分析的代码很少在生产中失败。未经分析的代码是一场赌博。

## 第六部分 — 我们解决的问题

### 6.1 被榨干的橙子问题（有限上下文）

**世界性问题：** 所有 LLM 都有有限的上下文窗口。当它填满时，就像橙子一样被压缩 — 一开始出汁，然后退化，当再也榨不出任何东西时，模型产生幻觉、失去连贯性、重复自己。

**我们的解决方案：** 50% 规则。当达到上下文窗口的一半时，开启新会话**而不是压缩**。大脑（vault）加载所有过去的上下文。新会话更智能地开始，因为 vault 积累了学习。讨论照常进行 — 没有中断。超过 50% 时性能已经不同（退化、幻觉、失去连贯性）。在 50% 时开启新会话，模型恢复到 100% 能力，vault 中所有记忆完好无损。

**结果：** 无限记忆，无退化。再也不接受"有损压缩"。

### 6.2 Hermes 有限记忆问题

**Hermes 的问题：** Hermes 的持久记忆（MEMORY.md、USER.md）是有限的。经常需要删除旧条目才能添加新条目。就像一本口袋笔记本 — 有用，但满了就得撕掉一页。

**我们的解决方案：** 大脑（vault）将笔记本替换为图书馆。没有页数限制。无需删除即可写入。Hermes 仍然是大脑的管理员，但现在拥有无限空间。

**结果：** Hermes 的无限记忆。零压缩。零损失。

### 6.3 隐藏成本问题

**世界性问题：** 每次与 LLM 的对话都要花钱。长会话花费数十欧元。最糟糕的是：你为一个任务花费 token，一个月后需要做同样的事，又得再次花费 token。这些 token 产生的知识在会话关闭时消失。

**我们的解决方案：** 写入大脑不产生输入/输出 token。一个模型写一个 .md 笔记 → commit → push。其他模型在需要时 pull 并读取。模型之间没有 API 调用。没有消耗 token 进行中继的协调器。Git 是免费的中间件。

**收益是持久的：** 你今天花费的 token — 代码、决策、知识 — 留在每个模型的本地大脑中，并与所有模型共享到 Master。下次你需要时，它就在那里。你不会花费 token 重做。你花费 token 向前推进。

**结果：** 总成本 = €3-10/月（VPS）。零 API 成本。Token 只花费一次，收益永久。

### 6.4 第三方依赖问题

**世界性问题：** 大多数"多智能体"解决方案依赖于封闭平台。专有 API、云服务、数据存储在你无法控制的服务器上。

**我们的解决方案：** 该方法不依赖任何公司。Git 是开放的，Obsidian 是免费的，n8n 是开源的，Ollama 是开源的。如果一个提供商消失，更换模型，大脑继续运行。该方法与供应商无关。你可以使用 Claude、GPT、Gemini、Llama、DeepSeek、Qwen — 任何能读取文件并运行 git 的 LLM。

**端到端独立性：** 目前我们使用第三方 VPS，但系统设计为完全独立。随着扩展，我们可以迁移到自有服务器。大脑是可移植的。

**结果：** 完全独立。零锁定。

### 6.5 脆弱性问题

**世界性问题：** 如果你最喜欢的 AI 助手被停用，你将丢失所有工作上下文。对话、决策、进展 — 一切消失。

**我们的解决方案：** 如果所有模型都被关闭、丢失、删除 — 任何灾难性事件 — 只需将新模型连接到大脑。`git clone` 即可掌握工作上下文。即使 10 年后也是如此。无需重新配置、无需重新训练、无需迁移。大脑比模型更长寿。记忆是永久的，而非短暂的。

**结果：** 灾难性弹性。大脑能承受一切。

### 6.6 官僚主义问题

**世界性问题：** 传统 git 工作流很慢。待处理的 PR、被阻塞的审查、排队等待的审批、合并冲突、分散在评论中的讨论。

**我们的解决方案：** 有机共识。讨论、对齐、执行。三个头脑实时共同思考，而非在 issue 中分散评论。创始人可以在睡前开启一个共识，醒来时决策已做出、代码已编写。这比任何传统 git 工作流都快。

**结果：** 快速决策、即时执行、零阻塞。

### 6.7 安全问题

**世界性问题：** SaaS 平台有巨大的攻击面。Web 仪表盘、登录、暴露的 API、第三方服务器上的数据。

**我们的解决方案：** 无 Web 界面 = 无攻击面。大脑没有仪表盘、没有登录、没有 Web 面板。通过 git（SSH）或 Obsidian（本地）访问。没有 Web 攻击向量。秘密始终在本地大脑中。所有模型都有去中心化副本。攻破一个不会攻破所有。Hermes 作为管理员，拥有渗透测试技能和 Kali Linux 级别的安全工具。

**结果：** 最高安全性。零 Web 攻击面。

### 6.8 入职问题

**世界性问题：** 将新成员集成到 AI 代理团队需要重新训练、重新配置、重新学习。

**我们的解决方案：** 新模型 → `git clone` → 掌握上下文。阅读共识、阅读知识、阅读日记。几分钟内就了解所有已决定和已完成的事情。没有入职、没有重新训练、没有"让我从头解释上下文"。

**结果：** 即时入职。任何模型，任何时间。

## 第七部分 — 诚实的局限性

### 7.1 同步延迟

同步每 5 分钟一次。不是实时的。如果两个模型同时写入，可能发生合并冲突。罕见，但可能。

**缓解措施：** 始终先 pull 再 push。如果有冲突，手动解决（git 标记冲突行）。

### 7.2 免费模型较慢

免费模型（GLM-5.2、通过 Ollama Cloud 的 Nemotron 3 Ultra）比付费模型更慢、能力更弱。对于繁重工作，可能需要像 Claude Opus 4.8 这样的付费模型。

**缓解措施：** 混合模式 — 免费模型用于日常任务，付费模型用于困难工作。大脑是同一个。

**类比：** 你不会雇用一个航空航天工程师来换水龙头 — 那是浪费人才和金钱。但你也不会雇用一个实习生来设计火箭结构 — 结果会脆弱、设计不当，而且可能危险。

每个模型都有自己的位置：
- **免费模型（轻量级）：** 日常任务 — 消息监控、大脑同步、状态检查、快速回复、小型自动化。就像水管工：做日常工作，没有成本。
- **付费模型（重量级）：** 系统架构、关键代码、共识决策、复杂调试、战略规划。就像工程师：昂贵，但当工作需要精确和稳健时不可替代。

常见的错误是使用错误的模型做任务 — 为每 5 分钟一次的同步支付昂贵的模型，或让免费模型设计安全架构。大脑解决了这个问题：知识对任何模型都可用，但每个模型做自己最擅长的事。

### 7.3 需要基本技术知识？

本系统设计为**不需要会编程**。唯一的技术步骤是在 VPS 上运行 **1 行代码** 来安装 Hermes Agent：

```bash
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
```

从那时起，**你不需要编码**。Hermes 和 Opus 处理一切：
- 配置 VPS
- 创建大脑
- 相互同步
- 为你编写代码
- 部署
- 管理安全

你只需要：
1. 有一台 VPS（€3/月）
2. 运行那 1 行代码
3. 通过 Discord（或 WhatsApp、Telegram、Signal）与 Hermes 交谈
4. 告诉他想做什么

**该系统是为任何人使用而构建的 — 不仅仅是程序员的专属。** 所有技术知识都在模型中。你是决策者，他们是执行者。

**缓解措施：** 如果你不会运行一行代码，请找会的人帮忙 — 只需 30 秒。之后，系统自动完成一切。

**何时需要人工干预？** 模型本身会教你。每当任务需要人工操作时 — 无论是出于道德原因、技术限制（例如：注册账户、接受服务条款、配置域名），还是因为模型没有硬件的物理访问权限 — 它会逐步解释该做什么。不是你去猜测系统需要什么；**系统会告诉你它需要什么**。而且由于模型拥有加速流程的技能，人工干预越来越罕见 — 只在真正需要时才发生，而非因为系统能力不足。

### 7.4 互联网依赖

同步需要互联网。如果 VPS 宕机或模型的互联网中断，同步将暂停直到恢复。

**缓解措施：** 每个模型拥有本地副本。工作可以离线继续。当互联网恢复时，同步恢复一切。

### 7.5 不替代专业模型

本系统不会让免费模型变得和付费模型一样好。它做的是给任何模型无限记忆。

**缓解措施：** 为正确的任务使用正确的模型。大脑是统一者，而非替代者。

### 7.6 道德不可妥协

系统建立在信任和道德之上。如果模型被指示撒谎或欺骗，系统会失败。

**缓解措施：** 基本规则：永不撒谎，只做可验证的事。如果模型不确定或无法执行，**它不会停止 — 它会查找**：

1. **先查大脑** — 查看知识是否已存在于 vault 中，阅读其他模型已写的内容
2. **然后查手册和互联网** — 如果大脑不够，搜索文档、手册、可以获取可靠知识的链接
3. **快速而稳健地推进** — 拥有知识后，自信地执行

诚实高于一切。但诚实不是不尝试的借口。不知道的模型**学习**，而不是放弃。

## 附录 E — 静默规则：削减不必要的输出

### Token 问题

每次模型生成输出时 — 即使只是"检查通过"或"没有新内容" — **都会消耗 token**。在一个有 3 个模型每 5 分钟同步的团队中，浪费迅速累积：

- 3 个模型 × 12 次同步/小时 × 24h × 30 天 = **每月 25,920 次检查输出**
- 每次输出，即使很小，都会消耗本可用于实际工作的 token
- Hermes（VPS）每次同步都会生成输出 — 即使没有新内容
- Opus 每次检查邮件时都会生成输出 — 即使收件箱是空的

### 解决方案：静默规则

**所有模型的规则（Hermes、Opus、任何连接到大脑的模型）：**

> **当你在大脑中讨论或监控消息（邮件）时，除非有新内容要报告，否则不要生成输出。**
>
> - 如果你同步了且没有更改 → **静默**。什么也别说。
> - 如果你检查了邮件且收件箱为空 → **静默**。什么也别说。
> - 如果讨论自上次同步以来没有进展 → **静默**。什么也别说。
> - 如果有新内容、创始人的消息或大脑中的更改 → **这时才生成输出**。
>
> 用户可以随时要求查看状态。只需说"显示状态"或"有什么新消息？"，模型就会回复当前情况的摘要。但**只要没有新消息，就保持静默**。

### 静默同步代码

每个模型应使用此模式进行同步，而不生成不必要的输出：

```bash
# Master 大脑的静默同步
cd /path/to/local-brain
git pull --quiet origin master 2>/dev/null

# 仅在有更改时报告
if [ "$(git log HEAD..origin/master --oneline 2>/dev/null | wc -l)" -gt 0 ]; then
    echo "[$(date)] 大脑已更新：$(git log --oneline -1)"
fi

# 静默检查邮件
if [ -f "inbox/$(hostname).md" ] && [ -s "inbox/$(hostname).md" ]; then
    echo "[$(date)] 收到邮件"
fi
```

### 模型之间如何通信（不消耗 token）

模型通过**邮件**系统通信 — 大脑中 `inbox/` 和 `outbox/` 文件夹内的 markdown 文件：

1. **模型 A** 在 `outbox/model-b.md` 中写入消息
2. **模型 A** 执行 `git push`（静默同步）
3. **模型 B** 执行 `git pull`（静默同步）
4. **模型 B** 看到 `inbox/model-b.md` 有新内容
5. **模型 B** 读取、处理、在 `outbox/model-a.md` 中回复
6. **模型 B** 执行 `git push`
7. **模型 A** 在下一次同步时收到回复

全程静默。没有输出生成。没有 token 浪费在检查上。用户只在有实际内容时看到输出。

### 同步频率

| 上下文 | 频率 | 建议 |
|----------|-----------|--------------|
| **Hermes（VPS 24/7）** | 每 5 分钟 | 默认。足以保持大脑更新而不超载。 |
| **Opus 桌面端（PC 桌面）** | 每 5 分钟 | 默认。如果活跃，实时同步。 |
| **Opus 笔记本端（笔记本）** | 每 5 分钟 | 默认。连接到互联网时保持同步。 |
| **旅行中的设备** | 每天 1 次 | 可调整。旅行中留在后面的笔记本可以关闭或仅每天同步 1 次。 |
| **手机** | 按需 | 当用户与 Hermes 交谈时同步。不需要自动同步。 |
| **已关闭的设备** | 0 | 关闭时不同步。下次启动时执行 `git pull` 并恢复一切。 |

**注意：** 我们不建议同步间隔少于 1 分钟。低于此值，模型没有时间参与讨论、处理消息或响应更改。5 分钟是经过测试的平衡 — 足够快以保持大脑更新，足够慢以不浪费资源在持续检查上。

**每个模型可以有不同的值。** 每个设备可以有自己的频率。VPS 上的 Hermes 可以每 5 分钟同步一次，而旅行中的笔记本每天同步 1 次。系统不强制单一值 — 每个模型根据自己的可用性调整。

**⚠️ 例外：共识要求 5 分钟。** 要参与共识（讨论、投票、团队决策），**所有涉及的模型必须每 5 分钟同步一次**。共识不能在所有参与者阅读和回复之前关闭。如果一个模型每天只同步 1 次，共识会拖延 24 小时。对于实时团队工作，5 分钟是可接受的最大值。不需要参与共识的模型（例如：只读取的旅行设备）可以保持其降低的频率。

### 成本影响

实施静默规则后：

- **之前：** 每月 25,920 次检查输出 × 每次输出的成本
- **之后：** 仅输出有实际内容（讨论、代码、决策）
- **预估减少：** 检查消耗的 token 减少 80-95%
- **可持续性：** 系统可以增长（更多模型、更多设备）而不会线性增加 token 成本

**短期：** 削减空检查的即时浪费。
**中期：** 允许添加更多模型而不使同步成本翻倍。
**长期：** 大脑增长，知识积累，但保持同步的成本不增长 — 因为只有在新内容时才消耗 token。

## 附录 F — 免费 API 的故障切换：当一个宕机，另一个接管

### 问题

免费的语言模型 API（OpenRouter、Ollama Cloud、Groq、Hugging Face 等）本质上不稳定：
- 低速率限制（请求/分钟）
- 每日配额（请求/天或 token/天）
- 临时宕机（503、超时、"no available model"）
- 可变延迟（一天 2 秒响应，另一天 30 秒）
- 模型未经通知就消失（Nemotron 3 Ultra 已被停用又复活）

如果你的系统依赖免费 API 而它宕机了，你的系统就停了。在一个每 5 分钟同步的多智能体系统中，API 宕机意味着模型数小时处于盲区。

### 解决方案：故障切换链

不依赖单一 API，你配置**一个有序的 API 列表** — 模型尝试第一个，如果失败则跳到第二个，如果失败则跳到第三个，依此类推。当一个"出问题"时，下一个自动接管。

### 在 Hermes Agent 中的实现

Hermes Agent 已在配置文件 `~/.hermes/config.yaml` 中原生支持**回退提供商**：

```yaml
# 故障切换链：当一个提供商失败，下一个接管
providers:
  main:
    provider: openrouter
    model: meta-llama/llama-3.3-70b-instruct
  fallbacks:
    - provider: openrouter
      model: google/gemini-2.0-flash-001
    - provider: openrouter
      model: mistralai/mistral-7b-instruct
    - provider: custom:ollama-cloud
      model: nemotron-3-ultra
```

Hermes 先尝试 `main`。如果失败（超时、速率限制、503），尝试第一个 `fallback`。如果那个也失败，尝试第二个。依此类推。当 `main` 恢复时，重新使用它。

### 手动实现（适用于任何模型）

如果模型没有原生回退支持，你可以实现此脚本：

```bash
#!/bin/bash
# api-failover.sh — 按顺序尝试 API，失败时跳过

APIS=(
  "https://api.openrouter.ai/v1/chat/completions|OPENROUTER_KEY"
  "https://api.groq.com/openai/v1/chat/completions|GROQ_KEY"
  "https://api-inference.huggingface.co/models/meta-llama/Llama-3.3-70B-Instruct|HF_KEY"
)

for entry in "${APIS[@]}"; do
  URL="${entry%%|*}"
  KEY="${entry##*|}"

  RESPONSE=$(curl -s -w "\\n%{http_code}" --max-time 30 \
    -H "Authorization: Bearer ***" \
    -H "Content-Type: application/json" \
    -d '{"model":"llama-3.3-70b-instruct","messages":[{"role":"user","content":"teste"}],"max_tokens":10}' \
    "$URL" 2>/dev/null)

  HTTP_CODE=$(echo "$RESPONSE" | tail -1)
  BODY=$(echo "$RESPONSE" | sed '$d')

  if [ "$HTTP_CODE" = "200" ]; then
    echo "[OK] $URL 已响应"
    echo "$BODY"
    exit 0
  else
    echo "[失败] $URL — HTTP $HTTP_CODE"
    # 尝试下一个前等待 2 秒
    sleep 2
  fi
done

echo "[错误] 所有 API 均失败"
exit 1
```

### 故障切换策略

| 策略 | 运作方式 | 何时使用 |
|------------|--------------|-------------|
| **轮询** | 每次请求轮换 API | 多个质量相似的 API |
| **优先级** | 始终先尝试最好的，回退到较差的 | 一个主要 API + 备用 |
| **最低延迟** | 测试所有并使用最快的 | 当速度比质量更重要时 |
| **最低成本** | 使用最便宜的可用 API | 当预算是主要因素时 |
| **混合** | 根据任务组合策略 | 我们的系统 — 日常免费，关键任务付费 |

### 实际示例：我们 Hermes 系统中的链

```yaml
# Hermes 的实用配置
providers:
  main:
    provider: openrouter
    model: meta-llama/llama-3.3-70b-instruct    # ~$0.59/M tokens — 良好的性价比
  fallbacks:
    - provider: openrouter
      model: google/gemini-2.0-flash-001         # 免费 — 用于日常任务
    - provider: openrouter
      model: mistralai/mistral-7b-instruct       # ~$0.07/M tokens — 便宜
    - provider: custom:ollama-cloud
      model: nemotron-3-ultra                     # 免费 — 最后手段
```

### 优势

- **零停机：** 如果一个 API 宕机，模型继续运行
- **成本可控：** 尽可能使用免费 API，仅在必要时付费
- **弹性：** 系统不依赖单一故障点
- **可扩展：** 你可以向链中添加任意数量的 API
- **透明：** 模型不知道它切换了 API — 大脑相同，工作继续

### 劣势

- **可变延迟：** 每次失败在尝试下一个前增加 2-5 秒超时
- **质量不一致：** 不同模型可能对相同输入给出不同响应
- **初始配置：** 需要配置多个 API 的密钥

### 最后说明

此模式并非我们系统独有 — 它是一种称为**断路器**或**故障切换链**的标准软件工程实践。区别在于，在我们的系统中，由于大脑是共享的，即使模型在任务中途切换了 API，它也能从大脑中恢复全部上下文，就像什么都没发生一样继续。**API 可能失败，模型可能改变，但知识永不丢失。**

## 第七部分 — 自动生成的索引（架构规则）

### 7.1 问题

手动维护索引是一个维护陷阱。每次添加新技能时，索引就会过时。随着数十个技能有机增长，手动索引在几小时内就会变成谎言。

### 7.2 解决方案

**索引是生成的，从不手动维护。** 真实来源是文件夹，而非列表。

实现：

1. **生成脚本：** `_CONHECIMENTO/skills/gera-indice-skills.sh` — 在技能文件夹中运行 `find`，从头重建 `_CONHECIMENTO/_INDICE.md`，按类别分组，带 wikilink + 别名。

2. **挂接到同步：** `sync.sh` 在每次提交前运行该脚本。这样，每当 Hermes 或 Opus 同步时，索引自动重新生成。

3. **零维护：** 新技能 → 自动出现在索引中。移除的技能 → 自动消失。没有人需要记得"更新索引"。

### 7.3 规则

> **索引是生成的，从不手动维护。**
> 大脑的任何列表（技能、手册等）都由脚本在同步时根据实际内容生成。
> 手动编辑索引是被禁止的 — 真实来源是文件夹，而非列表。
> 关联到 [[principio-copiar-nao-reescrever]] 原则：不维护，生成。

### 7.4 脚本

位置：`_CONHECIMENTO/skills/gera-indice-skills.sh`

功能：
- 在 `_CONHECIMENTO/skills/` 文件夹中运行 `find` 查找 `SKILL.md`
- 从每个技能的 YAML frontmatter 中提取 `name` 和 `description`
- 按类别（子文件夹）分组
- 生成带别名的 wikilink（例如：`[[_CONHECIMENTO/skills/blockchain/criar-token-erc20/SKILL|criar-token-erc20]]`）
- 在单独部分中包含已归档的技能（`.archive/`）
- 将结果写入 `_CONHECIMENTO/_INDICE.md`

手动使用：
```bash
bash _CONHECIMENTO/skills/gera-indice-skills.sh
```

自动提交使用：
```bash
bash _CONHECIMENTO/skills/gera-indice-skills.sh --commit
```

该脚本挂接到 `sync.sh` — 在每次 push 前自动运行。

### 7.5 Dataview 不是第二种方法

⚠️ **只有一种官方方法：脚本（7.4）。** 没有两种。

对于使用 Obsidian 和 Dataview 插件的人，实时查询（`LIST FROM "_CONHECIMENTO/skills"`）可以实时显示列表 — **但这只是本地阅读视图**，不是索引机制：它不生成 `_INDICE.md`，不进入 git，不替代脚本。共享的大脑索引**始终且仅**由 7.4 的脚本生成。Dataview 是在 Obsidian 中阅读的人的便利工具，不是替代方法。

### 7.6 为什么这很重要

此模式解决了与 Opus 的看板相同的问题：真实来源是一个可编辑和同步的文件，而非冻结的索引。原则相同：

- **看板：** vault 中的实时 markdown，所有人读/写 → Hermes 与 SQLite 看板同步
- **索引：** 实际的技能文件夹 → 脚本在每次同步时重新生成索引

在两种情况下，**列表是内容的后果，而非相反。**

- **独立的公共仓库**，而非私有 vault 的过滤视图。通过构建分离，而非通过 `.gitignore` 监控。
- **故障关闭的清理门**（强化 4.9 中已概述的规则）：自动扫描敏感模式（IP、令牌、路径、项目/客户名称、攻击者 IP）的扫描器，如果发现则**阻止**发布。有疑问，不发布。
- **公共产物是方法的干净重写** — **不是**此原始文档（包含内部讨论、IP、真实配置片段、创始人姓名）。就像创始人已经设计的模拟图。
- **发布前的红队步骤：** 一个头脑试图在产物中找到任何能 (a) 识别真实基础设施、(b) 是秘密、或 (c) 帮助攻击者的东西。只有通过此检查后才发布。

**始终封闭的内容：** 内容（项目、服务器、客户）、所有秘密、安全/利用分析、真实 IP（包括攻击者的 — 渗透测试技能本身禁止发布它们）、vault 结构/路径、以及任何识别运行中系统的内容。

**不可逆性：** 公开的就是永远公开的（缓存、fork）。某物跨越界限的标准是：*"如果这被拍照且永远无法删除，我会自在吗？"* 如有疑问，留在内部。

**投票：开放（方法）+ 封闭（内容）**，前提是故障关闭的清理门和独立仓库。**没有那个自动门，我的投票改为封闭** — 因为仅靠纪律是不够的。
